# Compound Products Requirement

## 1. Problem Statement

The system currently supports selling direct stock products only.
Business users also need to sell **final products/services** that are composed of multiple inventory ingredients (for example: cake consumes flour, sugar, eggs).

At checkout, users should add only the final sellable item to cart. The system must automatically discount the defined ingredient quantities from inventory.

## 2. Goal

Enable creation and sale of compound items (recipe-based products or services) that automatically consume component stock during checkout in a single atomic transaction, with full movement traceability from sold parent item to all ingredient deductions.

## 3. Scope

In scope:

- Define sellable item types: stock item, compound item, service.
- Define recipe/BOM (bill of materials) for a compound item.
- Checkout expansion logic that converts sold compound quantity into ingredient stock deductions.
- Inventory policy allowing ingredient stock to become negative when there is insufficient quantity.
- Auditable inventory movement records linking compound sale to ingredient deductions.

Out of scope (phase 1):

- Multi-level recipes (compound uses another compound).
- Partial checkout / backorder behavior.
- Automatic unit conversion across incompatible units.
- Wastage/yield percentages.

## 4. Business Rules

1. A cashier can add only the final item to the cart.
2. During checkout, the system resolves the final item recipe and deducts each ingredient.
3. Recipe deduction formula:

   `ingredient_deduction = sold_quantity * ingredient_quantity_per_output`

4. Checkout must proceed even when one or more ingredients become negative.
5. Stock deductions and sales records must be committed in one Firestore transaction.
6. Service items with no stock impact are allowed and do not deduct inventory.
7. Recipe updates affect new checkouts only and do not mutate historical sales.
8. Logs must include both the sold parent line (stock/service/compound) and each sub-product movement generated by compound consumption.

## 5. Data Model Requirement

### 5.1 Sellable Item Classification

Add an item classification field (new or equivalent):

- `item_type: "stock" | "compound" | "service"`

Expected behavior:

- `stock`: direct inventory item, existing behavior.
- `compound`: recipe-driven item that consumes ingredients.
- `service`: sellable line item with no inventory deduction.

### 5.2 Recipe/BOM Structure

Add recipe definition for compound items (collection or subcollection):

- `compound_product_id`
- `version` (optional, recommended)
- `is_active`
- `ingredients[]`
  - `product_id` (must reference stock item)
  - `quantity_per_output`
  - `measurement_unit`

Constraints:

- Ingredient product must belong to same `org_id`.
- Ingredient cannot reference the same product as parent compound.
- Ingredient list cannot be empty for `compound` items.

### 5.3 Precision Requirement

Mass/volume ingredient deductions must support sub-unit precision (example: 500 g).

To stay consistent with current integer stock arithmetic, stock quantities for mass/volume must be stored in base minor units:

- mass: grams
- volume: milliliters
- unit: unit count

This keeps transaction math integer-safe and avoids floating-point errors.

## 6. Checkout Requirement

Given cart lines with mixed item types, checkout must:

1. Read all sold items.
2. Expand compound lines into ingredient deductions.
3. Aggregate deductions by `ingredient product_id`.
4. Do not block checkout on insufficient ingredient stock; resulting stock can be negative.
5. Update ingredient stock balances.
6. Write sale records for final sold items.
7. Write movement records that link each ingredient deduction to the parent sale line.

Required movement log record for traceability:

- `sale_inventory_movements`
  - `org_id`
  - `sale_id`
  - `sale_line_id`
  - `movement_group_id` (correlation id for one parent sale line and its generated movements)
  - `movement_type: "sale_parent" | "compound_consumption" | "service_sale"`
  - `product_id` (affected item)
  - `product_name_snapshot`
  - `product_sku_snapshot`
  - `parent_product_id` (final sold item id; null for root parent movement)
  - `parent_item_type: "stock" | "compound" | "service"`
  - `quantity_sold` (for parent movement)
  - `quantity_delta_stock` (negative for stock consumption, zero for service-only movement)
  - `measurement_unit`
  - `recipe_version_snapshot` (for compound-generated movements)
  - `stock_before` (optional, recommended)
  - `stock_after` (optional, recommended)
  - `moved_by`
  - `moved_at`

Minimum expected logging per cart line:

- `stock` item line: one `sale_parent` movement with negative `quantity_delta_stock`.
- `compound` item line: one `sale_parent` movement plus one `compound_consumption` movement per ingredient.
- `service` item line: one `service_sale` movement with `quantity_delta_stock = 0`.

## 7. Security and Authorization Requirement

1. Only manager/admin can create or modify recipe definitions.
2. Cashier can execute checkout only.
3. Checkout may update stock only via allowed stock mutation path.
4. Movement logs are append-only.

## 8. UI/UX Requirement

1. In add/edit item flow, user can choose item type.
2. If item type is `compound`, show recipe builder:
   - search ingredient product
   - set quantity per output
   - set unit
3. Cart UI still shows only final item.
4. If checkout would drive ingredient stock below zero, allow checkout and show a warning with required vs available quantity.

## 9. Acceptance Criteria

1. Given a compound "Cake" with ingredients flour 500 g and sugar 500 g, when cashier sells 1 Cake, then flour and sugar each decrease by 500 g.
2. Given same recipe, when cashier sells 3 Cakes, then flour and sugar each decrease by 1500 g.
3. Given insufficient flour stock, checkout succeeds and flour stock becomes negative by the shortage amount.
4. Given cart has direct stock items + compound items + services, checkout applies correct deductions only for stock and compound ingredients.
5. Historical sales remain reproducible even after recipe changes (versioned or snapshotted recipe in sale context).
6. All writes in checkout are atomic: either all stock deductions + sales + movement logs persist, or none do.
7. Given one compound line is sold, logs contain one parent movement plus all generated ingredient movements linked by `sale_line_id` and `movement_group_id`.
8. Given one service line is sold, logs contain a service movement entry with `quantity_delta_stock = 0`.

## 10. Suggested Implementation Phases

Phase 1:

- Add item type and single-level recipe model.
- Add checkout expansion + aggregated ingredient deduction.
- Add negative-stock checkout policy and user-facing warnings.

Phase 2:

- Add recipe versioning snapshot in sales records.
- Add movement analytics and reporting.

Phase 3:

- Add advanced manufacturing concepts (yield, wastage, substitutions).
