# Sales Flow Schema and Rules Notes

## Firestore Writes Used by Sales Checkout

Checkout transaction currently writes:

1. `products/{productId}`

- Decrement `current_stock`
- Update `updated_by`
- Update `updated_at`

2. `sales/{saleDocId}` (one document per cart line)

- `org_id`
- `sale_id`
- `client_txn_id`
- `product_id`
- `sku`
- `quantity`
- `unit_price`
- `total_amount`
- `sold_by`
- `sold_at`
- `created_at_client`
- `device_id`
- `cart_id`
- `cart_label`

## Target Write Model for Compound and Service Traceability

To support compound products/services with full movement auditability, checkout should also write:

3. `sale_inventory_movements/{movementId}` (one or more docs per sale line)

- `org_id`
- `sale_id`
- `sale_line_id`
- `movement_group_id` (groups parent line movement with generated sub-movements)
- `movement_type` (`sale_parent` | `compound_consumption` | `service_sale`)
- `product_id` (item affected by this movement)
- `product_name_snapshot`
- `product_sku_snapshot`
- `parent_product_id` (null for root parent movement)
- `parent_item_type` (`stock` | `compound` | `service`)
- `quantity_sold` (for parent movement)
- `quantity_delta_stock` (negative for stock deduction, zero for service movement)
- `measurement_unit`
- `recipe_version_snapshot` (when generated by compound recipe)
- `stock_before` (optional)
- `stock_after` (optional)
- `moved_by`
- `moved_at`

Expected movement shape by cart line type:

- Stock line: one `sale_parent` movement.
- Compound line: one `sale_parent` movement + one `compound_consumption` movement per ingredient.
- Service line: one `service_sale` movement with `quantity_delta_stock = 0`.

## Security Rule Notes

`firestore.rules` was updated so cashier role can update products only for stock deduction use cases.

Cashier product updates are now restricted to changing only:

- `current_stock`
- `updated_at`
- `updated_by`

Additional guards:

- `current_stock` must stay integer and non-negative.
- `current_stock` must not increase in cashier path.
- `updated_by` must equal authenticated user UID.

Manager/admin paths remain available for broader product updates.

## Index Notes

Current lookup query in `services/sales.ts` uses:

- `where("org_id", "==", orgId)`
- `where("is_active", "==", true)`
- `orderBy("updated_at", "desc")`

This relies on the existing composite index documented for `products`.

Sales documents are written by deterministic IDs and currently not queried by composite filters in this implementation.
